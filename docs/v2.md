# wasm-compiler v2 Roadmap

## Status at v1

- [x] x86-64 code generation (integer ops, loads, stores, control flow)
- [x] Linux inline syscalls via `platform.System` (no libc, no linker relocation)
- [x] Pointer translation: wasm linear-memory offset → native absolute address
- [x] `memory.fill` via `rep stosb`
- [x] ELF output via linker
- [x] Bug fix: unconditional R15 translation for syscall pointers (offset 0 valid)

---

## v2 Planned Updates

### 1. `memory.copy` — `rep movsb`

**File:** `memory.go`  
**Status:** currently returns `fmt.Errorf("memory.copy not implemented")`

The direct counterpart to the existing `memory.fill` / `rep stosb` implementation.

Wasm stack before: `[..., dst, src, n]` (n is TOS)

```
emitPopR(RCX)   // n   → rcx (count)
emitPopR(RSI)   // src → rsi (source, needs R15 translation)
emitPopR(RDI)   // dst → rdi (destination, needs R15 translation)

// zero-extend i32 wasm values
mov ecx, ecx
mov esi, esi
mov edi, edi

// translate to native addresses
add rsi, r15
add rdi, r15

// copy (Direction Flag is clear per SysV ABI)
rep movsb       // F3 A4
```

Note: unlike syscall pointers, both src and dst here must use the NULL guard
(`emitSafePointerTranslate`) since a wasm program could legitimately copy
from/to a zero-offset buffer — actually no, same problem as the write fix.
Use unconditional `emitAddR15To` for both, consistent with the v1 fix.

---

### 2. `linux::exit_group` syscall

**File:** `main.go` (sample), `platform/linux/system/`  
**Status:** process exit is currently implicit — whatever `main` returns flows
through the normal function epilogue and the OS picks up whatever was in rax.

Add an explicit `exit_group` import so the wasm module controls its own exit
code intentionally:

```go
// In buildModule():
tExit := m.Types.AddFuncType(wasm.FuncType{
    Params: []wasm.ValType{wasm.I32}, // exit code
})
m.Imports.AddFunc("linux", "exit_group", tExit)
```

Linux syscall number (x86-64): **231**  
Signature: `(ParamInt) -> []` (no result)

In `codeMain()`:
```go
b.I32Const(0).Call(fnExitGroup)  // explicit clean exit
```

This also exercises the zero-result syscall path in `emitInlineSyscall` (the
`if len(ft.Results) > 0` branch is skipped, rax is discarded).

---

### 3. `call_indirect`

**File:** `control.go`  
**Status:** `emitCallIndirect` returns `fmt.Errorf("call_indirect not yet supported")`

Required for function pointers, vtables, and most non-trivial wasm output from
real toolchains (clang, wasi-sdk, etc.).

Implementation sketch:

1. Pop the `i32` table index from the wasm stack into a scratch register.
2. Bounds-check against the table length (ud2 / trap on failure).
3. Load the function pointer from the table (stored in `obj.Data` in the
   globals reserve region, or as a separate table section in the object).
4. Type-check: compare the `typeIdx` immediate against the type stored
   alongside the function pointer in the table entry.
5. Emit an indirect call: `call rax` (FF D0).

The table itself needs a representation in `object.WasmObj` and linker support
to patch absolute addresses into it at link time — this is the largest piece of
new infrastructure in v2.

---

### 4. BSS segment in the linker

**File:** `linker/` (ELF output)  
**Status:** the full `obj.Data` buffer (up to 1 MB+ of zeroes) is written
literally into the ELF file, producing ~3 MB binaries for trivial programs.

Split `obj.Data` into:
- **`.data`** — the initialized portion (data segments + globals reserve)
- **`.bss`** — the remaining zero-initialized pages

ELF `.bss` has no file content; the OS zero-fills it on load. A hello-world
binary should be in the low tens of KB, not 3 MB.

Concrete change: in `compile.go`, track the highest written byte in `obj.Data`
and expose it as `obj.DataInitEnd`. The linker uses this as the `.data`/`.bss`
boundary.

---

### 5. Floating-point (SSE2)

**Files:** `body.go`, `arith.go`, new `float.go`  
**Status:** all f32/f64 opcodes return `fmt.Errorf("not implemented")`

SSE2 is baseline on every x86-64 CPU. The encoding patterns are mechanical
once the integer side is established.

Priority order:

| Group | Opcodes | Notes |
|---|---|---|
| f32/f64 loads & stores | `0x2A, 0x2B, 0x38, 0x39` | `movss`/`movsd` with existing SIB encoding |
| f32 ↔ f64 const | `0x43, 0x44` | already handled as raw i32/i64 bits on the stack |
| f32/f64 add/sub/mul/div | `0x92–0x9A` | `addss`, `subss`, etc. |
| f32/f64 comparisons | `0x5B–0x66` | `ucomiss`/`ucomisd` + `sete`/`setb` etc. |
| int ↔ float conversions | `0xA8–0xBF` | `cvtsi2ss`, `cvttss2si`, etc. |

Stack convention: f32/f64 values can be kept on the integer stack as raw bits
(the same 8-byte slots), moved into XMM registers only for arithmetic, then
pushed back. This avoids introducing a separate float stack or register
allocator for v2.

### 6. `memory.init` + `data.drop`

**File:** `body.go` (`dispatchFC`, cases 8 and 9)  
**Status:** `data.drop` is already a no-op (correct — data is inlined at
startup). `memory.init` returns an error.

`memory.init` copies a slice of a passive data segment into linear memory at
runtime. Since passive segments are already copied into `obj.Data` during
compilation, `memory.init` can be emitted as a bounded `rep movsb` with
compile-time source address and length — no runtime segment table needed.

Depends on `memory.copy` (item 1) being implemented first.
